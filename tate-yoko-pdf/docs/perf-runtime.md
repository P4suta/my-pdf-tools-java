## Peak memory vs input size (warm median)

Peak RSS = a fixed **~115 MiB JVM/PDFBox/AWT floor** + the document held in
heap. The floor dominates small inputs (so `RSS/size` looks huge and then
falls); for large inputs the marginal RSS settles to **roughly the input
size** ‚Äî that is the output document's memory-only stream cache accumulating
every cloned page stream until save. Extrapolating that slope against the
launcher heap locates the input size at which it would OOM: the cliff this
benchmark exists to find. (Peak RSS is sampled from `/proc`, so a very short
run can read slightly under the true peak.)

## Memory safety

Two defenses keep the heap from being exhausted by a very large scan:

1. **RAM-proportional heap** ‚Äî the launcher runs with
   `-XX:MaxRAMPercentage=75.0` (see `app/build.gradle.kts`), so the heap
   scales to the host and honors container cgroup limits instead of a fixed 2g.
2. **`--low-memory`** ‚Äî spills cloned page streams to a temp file
   (`java.io.tmpdir`) via PDFBox's scratch stream cache, bounding heap
   regardless of input size, at the cost of some disk I/O.

Verified on a large input under a deliberately tiny heap: with
`JAVA_TOOL_OPTIONS=-Xmx64m` the default (in-memory) conversion exits 137
(OutOfMemory), while the same conversion with `--low-memory` completes
successfully and produces an identical page count. Note that if
`java.io.tmpdir` is a tmpfs (RAM-backed), `--low-memory` does not free
physical memory ‚Äî point it at real disk on memory-constrained hosts.
